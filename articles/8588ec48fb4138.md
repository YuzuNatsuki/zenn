## なぜ「声色」で NPC の反応を変えたいのか

ゲームや対話 AI では、プレイヤーの**発言内容**に NPC が反応するのが普通です。でも現実の会話では、同じ言葉でも**声のトーン**で相手の反応は変わります。

> 「大丈夫？」と明るく聞くのと、低い声で聞くのと、受け取り方は全然違う。

既存の LLM チャットボットはテキストしか見えないので、この「声色のニュアンス」を拾えません。AmiVoice の感情分析（ESAS）なら、**テキスト認識と同時に 20 種類の声色パラメータを取得**できます。これを Claude に渡すことで、

```
同じテキスト × 違う声色 → 違う NPC 反応
```

という、テキストだけの LLM では実現できない体験を作れる、というのが今回のアプローチです。

## 作ったもの：「声のペルソナ」

音声ファイルをアップロードすると、4 人の NPC キャラが**声色を踏まえて**返答するブラウザアプリです。

### デモ：同じセリフ × 4 通りの声色

4 通りの声色で録音してアップロードすると、キャラクターの返答が全く変わります。

| 声色 | 検出されたスコアの特徴 | Claudeの返答 |
|------|----------------------|-------------|
| 沈んだ声でゆっくり | `upset > 0`, atmosphere マイナス | 「ねえ、声、震えてるよ。無理しなくていいから、ちゃんと話して」 |
| 怒り混じりに速く | `aggression > 0` | 「いやちょっと、誰に怒ってんのよ。落ち着きなって」 |
| 軽快に笑いながら | `content > 0`, energy 高め | 「ふーん、楽しそうじゃん？でも本当はそうでもないんでしょ？」 |
| 落ち着いて朗読 | 補助指標ほぼ 0 | 「ふーん、なんかちゃんと考えてるんだね。続き聞かせて」 |

**テキストは完全に同じ**でも、声色だけでLLMの理解と返答が変わる — これが今回作りたかった体験です。

### システム構成

```
┌─────────────┐  音声ファイル  ┌──────────────┐  multipart  ┌─────────────────┐
│  ブラウザ    │ ───────────→ │  Node サーバ  │ ──────────→ │  AmiVoice API   │
│  (vanilla JS)│              │  (Express)    │             │ 非同期HTTP v1   │
│             │              │               │ ←─ポーリング─│ 認識+感情分析   │
│             │              │               │             └─────────────────┘
│             │              │               │
│             │              │               │ JSON       ┌─────────────────┐
│             │ ←─ JSON ───── │               │ ──────────→│ Claude API      │
└─────────────┘              └──────────────┘             │ (Sonnet 4)      │
                                                          └─────────────────┘
```

| レイヤ | 技術 | 役割 |
|--------|------|------|
| フロント | Vanilla JS | 音声アップロード、UI 更新 |
| サーバ | Node.js + Express | API キー隠蔽、AmiVoice / Claude へのプロキシ |
| 音声認識・感情分析 | AmiVoice 非同期 HTTP API v1 | テキスト認識 + 20 種の感情パラメータ |
| 対話生成 | Claude API (`claude-sonnet-4-20250514`) | キャラクター口調の NPC 返答生成 |

**なぜ非同期 HTTP API なのか？**  
感情分析（`sentimentAnalysis=True`）は **WebSocket API では使えず、非同期 HTTP API v1 でしか取得できない**ためです。v2 でも 2026 年 7 月までは非対応とのこと。

## 実装の全体フロー

```
1. ユーザーが音声ファイルをアップロード
2. POST /api/recognize
   → AmiVoice 非同期 HTTP API にジョブ投入
   → ポーリングで completed まで待つ
   → 認識テキスト + 20 種の感情スコアを返す
3. フロントで話速・口調を分析
4. POST /api/chat
   → テキスト + 感情スコア + キャラクターの情報を Claude に渡す
   → NPC の返答（JSON）を返す
5. UI に キャラクターのセリフ・ステータスを反映
```

以下、実装で詰まった 4 つのポイントを **症状 → 原因 → 解決策** の形式で解説します。

---

## ハマりポイント 1：非同期 API はポーリング必須

### 症状

音声ファイルをアップロードしても、認識結果が常に `"..."` になる。

```json
{ "sessionid": "019e7720cacb...", "text": "..." }
```

「あれ？認識結果が `"..."`？」と何度も録音し直しました。

### 原因

AmiVoice 非同期 HTTP API の**ジョブ投入レスポンスの `text` はプレースホルダー**です。公式ドキュメントに明記されています：

> 成功時のレスポンスには sessionid が含まれています。これは、ユーザーの音声認識リクエストに対するジョブの ID で **ジョブの状態を確認したり、結果を得るために利用します**。 **`text` は常に "..." を返します**。

つまり、投入レスポンスをそのまま結果として返していたのが原因でした。

### 解決策

正しいフローは **投入 → ポーリング → 結果取得** の 3 ステップです。

```javascript
// ① ジョブ投入 → sessionid を取得
const submitRes = await fetch("https://acp-api-async.amivoice.com/v1/recognitions", {
  method: "POST",
  body: multipartBody, // u=APIキー, d=認識設定, a=音声ファイル
});
const { sessionid } = await submitRes.json();

// ② completed まで GET でポーリング
const POLL_INTERVAL_MS = 3_000;
const MAX_WAIT_MS = 5 * 60 * 1000; // 最大 5 分

while (true) {
  await sleep(POLL_INTERVAL_MS);
  const pollRes = await fetch(
    `https://acp-api-async.amivoice.com/v1/recognitions/${sessionid}`,
    { headers: { Authorization: `Bearer ${process.env.AMIVOICE_API_KEY}` } },
  );
  const data = await pollRes.json();

  if (data.status === "completed") break;  // ← ここで初めて text と sentiment が入る
  if (data.status === "error") throw new Error(data.error_message);
  // queued / started / processing → 続けてポーリング
}

// ③ completed のレスポンスから結果を取り出す
const text = data.text;
const sentiment = data.sentiment_analysis?.segments;
```

ステータス遷移は `queued → started → processing → completed`。`queued` から `started` に進むだけで**数十秒〜数分**かかることがあるので、タイムアウトは 5 分以上を確保してください。

:::details ジョブ投入時の multipart ボディ構築
Node.js 標準機能のみで multipart を手動構築しています（外部ライブラリ不要）。

```javascript
const boundary = "----AmiVoiceBoundary";
const dValue = "grammarFileNames=-a-general sentimentAnalysis=True loggingOptOut=True";

const bodyParts = [
  `--${boundary}\r\nContent-Disposition: form-data; name="u"\r\n\r\n${API_KEY}`,
  `--${boundary}\r\nContent-Disposition: form-data; name="d"\r\n\r\n${dValue}`,
  `--${boundary}\r\nContent-Disposition: form-data; name="a"; filename="audio"\r\nContent-Type: ${mimeType}\r\n\r\n`,
];
const bodyPrefix = Buffer.from(bodyParts.join("\r\n") + "\r\n");
const bodySuffix = Buffer.from(`\r\n--${boundary}--\r\n`);
const multipartBody = Buffer.concat([bodyPrefix, audioBuffer, bodySuffix]);
```

`sentimentAnalysis=True` を `d` パラメータに含めることで、認識結果と同時に感情分析も取得できます。
:::

## ハマりポイント 2：値範囲がパラメータごとに違う

:::message
**Tips：パラメータ一覧は API で取得できる**  
20 個の感情パラメータの正式なキー名・値範囲は、公式が用意している以下のエンドポイントで取得できます。

```
GET https://acp-dsrpp.amivoice.com/v1/sentiment-analysis/ja/result-parameters.json
```

サーバー起動時にこの API を叩いて、自分のコードの定義とズレていないか検証するようにしておくと、API 仕様変更時にも気づけて安心です。
:::

### 症状

`upset > 55` で悲しみ判定を書いたが、**永遠に発動しない**。

### 原因

20 種類のパラメータには **4 種類の値範囲** があり、一律の閾値では判定できません。

| 範囲 | パラメータ | 特性 |
|------|-----------|------|
| **0〜100** | energy, stress, concentration, anticipation, intensive_thinking, brain_power | 比較的緩やかに変化 |
| **0〜30** | excitement, hesitation, uncertainty, imagination_activity, embarrassment, passionate, confidence, **aggression**, **upset**, **content**, dissatisfaction, extreme_emotion | **>0 で出ること自体が稀＝有意** |
| **1〜500** | emo_cog（感情バランス論理） | 65 前後がバランス |
| **-100〜100** | atmosphere（雰囲気会話傾向） | 負＝沈む / 正＝明るい |

特に 0〜30 範囲のパラメータは、公式ドキュメントにこう書かれています：

> **0 よりも大きな値を取ることはまれ** なので、0 よりも大きな値であれば高いとみなせます。

つまり `content`（喜び）の最大値は 30 なのに `joy > 50` で判定していたり、`upset`（最大 30）で `> 55` を期待していたりすると、**数学的に発動しない**ロジックになります。

### 解決策

0〜30 範囲のパラメータは **「>0 で出ているか」** で判定します。

```javascript
// ✅ 正しい判定ロジック
function sentimentToEmotion(s) {
  // 0-30 範囲：>0 で有意
  const isAngry = s.aggression > 0 || s.dissatisfaction > 0 || s.stress > 50;
  const isSad   = s.upset > 0 || s.atmosphere < -20;
  const isHappy = s.content > 0 || s.atmosphere > 30;

  if (isAngry) return "怒り気味";
  if (isSad)   return "悲しい";
  if (isHappy) return "上機嫌";
  return "普通";
}
```

Claude に渡すプロンプトでも、判定の優先順位を明示しています：

```
■ 判定の優先順位（高→低）
1. aggression>0 または dissatisfaction>0 または stress>50 → 怒り・苛立ち
2. upset>0 または atmosphere<-20 → 沈んだ気分・悲しみ
3. content>0 または atmosphere>30 → 上機嫌・喜び
4. 上記すべてに該当しない → 「普通の落ち着いた会話」として扱う（悲しいではない！）
```

## ハマりポイント 3：短い録音では energy 単独判定不可

### 症状

数秒の録音で、energy が 5〜6 前後に固定され、他のパラメータは全部 0。声色を変えてもスコアがほとんど変わらない。

| 録音 | energy | content | upset | aggression |
|------|--------|---------|-------|------------|
| 普通に話した | 5.5 | 0 | 0 | 0 |
| 早口で話した | 6.0 | 0 | 0 | 0 |
| 沈んで話した | 5.0 | 0 | 0 | 0 |

### 原因

公式ドキュメントによると、energy は「0 付近の値を取ることが多く、比較的緩やかに値が変化するパラメータ」で、0〜10 は「気落ち、退屈、疲れ、悲しみ」の範囲です。

**数秒の録音では energy 5 は「普通の落ち着いた会話」**であり、必ずしも「悲しい」とは言えません。コールセンター等の長時間会話で持続的に低い場合に意味があります。

### 解決策

2 つの対策を取りました。

**① 録音は 30 秒〜1 分以上にする**

300 文字程度のモノローグ（朗読時間 1 分前後）を作り、それを違う感情で録音すると、補助指標（0-30 範囲）がちゃんと反応するようになりました。

**② 短い録音では補助指標を最優先にする**

energy 単独で判断せず、0-30 範囲の補助指標（`aggression`, `upset`, `content`, `dissatisfaction`）が **>0 で出ているか** を最優先に判定します。補助指標がすべて 0 なら「普通の落ち着いた会話」として扱い、energy の低さだけで「悲しい」と決めつけません。

## ハマりポイント 4：Claude が指標名を口にする

### 症状

Claude の返答に「エネルギーが低いね」「ストレス値が高そう」のような**技術用語**が混ざる。

> 「なんだか元気ないじゃない。**エネルギーが低すぎる**わよ？」

### 原因

プロンプトに `energy=${sentiment.energy}` のような変数名をそのまま渡していたため、Claude がそれを会話に出して良いと解釈してしまいます。

### 解決策

**2 段構え**で制御しました。

**① システムプロンプトに禁止用語を明記**

```text
【絶対禁止：スコアや分析を漏らさない】
- 「エネルギー」「ストレス」「テンション値」「スコア」「数値」「分析」
  「指標」「値」「パラメータ」のような用語は一切使わない。

【スコアの正しい使い方】
- スコアは内部的な判断材料として使うだけで、返答には間接的にしか反映させない。
- 良い例：「声、震えてるよ」「なんだか沈んでるね」「ピリピリしてる感じ」
- 悪い例：「エネルギーが低いね」「ストレス値が高そう」
```

**② ユーザーメッセージのフォーマットを日本語名に変更**

```text
【声色データ（あなたの内部メモ／返答に絶対出さない）】
- 攻撃性: 0 / 動揺: 0 / 喜び: 0 / 不満: 0
- 活力: 5.5 / ストレス: 15 / 集中: 20 / 期待: 19
...
→ これらは「相手の声を聞いて感じ取った印象」として返答に滲ませる。
  数値も指標名も返答に出さず、人間らしい言葉で表現すること。
```

これで Claude の返答は完全に自然な日本語になりました。

## プロンプト設計のポイント

### システムプロンプトとユーザーメッセージの分離

Claude API の `cache_control` を使うため、**静的な部分（キャラクター設定）をシステムプロンプト**に、**動的な部分（NPC 状態・発話内容・感情スコア）をユーザーメッセージ**に分けています。

```javascript
const message = await anthropic.messages.create({
  model: "claude-sonnet-4-20250514",
  max_tokens: 512,
  system: [{
    type: "text",
    text: staticSystemPrompt,  // キャラ設定 + 感情解釈ルール（キャッシュ対象）
    cache_control: { type: "ephemeral" },
  }],
  messages: [{ role: "user", content: userMessage }],  // 毎回変わる情報
});
```

:::message
**NG パターン**：NPC 状態値をシステムプロンプトに埋め込むと、毎回内容が変わるのでキャッシュが機能しない上に、キャラクター切替後もキャッシュが残り古いキャラが返答することがあります。
:::

### Claude の JSON 出力を堅牢にパースする

Claude は時々 JSON を ```json ... ``` で囲んだり、整数に `+` 符号を付けたりします（JSON 仕様違反）。

```javascript
function parseNpcJson(rawText, character) {
  // 最初の { から最後の } までを抽出
  const start = rawText.indexOf("{");
  const end = rawText.lastIndexOf("}");
  let jsonStr = rawText.slice(start, end + 1);

  // "irritationDelta": +18 → "irritationDelta": 18 に修正
  jsonStr = jsonStr.replace(/:\s*\+(\d)/g, ": $1");

  return JSON.parse(jsonStr);
}
```

## こんなことができたら面白そう

今回は検証用のテストアプリなので、「とりあえず声色で反応が変わる」が確認できたところで止めています。ここから発展させると、面白そうなアイデアがいくつもありそうだと感じました。

### ゲーム / エンタメ系

- **ボイス RPG**: 戦闘前のセリフを「自信たっぷりに」言うと NPC が士気高く応戦、震えた声だとビビって逃げ出す、みたいな仕掛け
- **嘘発見 / 心理戦**: `imagination_activity` や `hesitation` が突出していたら NPC が「お前、本当のこと言ってないだろ？」と見抜く
- **ノベルゲーのアフレコ機能**: 自分の声でセリフを言うと、その感情に応じて分岐が変わる
- **声優育成シミュレーション**: 「悲しいシーンを悲しく言えているか」を `upset` / `atmosphere` で採点

### 実用 / 学習系

- **プレゼン練習コーチ**: 緊張で `hesitation` が高いとき「もう少し落ち着いて、間を取りましょう」とフィードバック
- **語学学習の発音 + 感情練習**: 「驚きを表現する英語フレーズ」の感情がちゃんと乗っているか採点
- **接客ロールプレイ**: 「お客様への謝罪」を録音して、誠実さ（`stress`/`atmosphere`）が伝わるトーンか診断
- **面接練習 AI**: 自信があるトーンか、不安が滲んでいるかを可視化してアドバイス

### コミュニケーション系

- **ぬいぐるみ AI**: 子供が悲しい声で話しかけたら優しく、嬉しそうな声なら一緒にはしゃぐ
- **メンタルヘルスログ**: 毎日 1 分の独り言録音から `energy` / `stress` の推移を蓄積し、不調の予兆を検知
- **議事録の感情マーカー**: 会議録音から「ここで誰かが不満そうだった」「ここで盛り上がった」を可視化

### 技術的に試したいこと

- **リアルタイム化**（v2 が感情分析対応したら WebSocket でリアルタイム反応にしたい）
- **TTS 連動**: NPC の返答を声優っぽい TTS で再生して、完全な音声会話に
- **3D アバター連動**: VRM モデルの表情・ジェスチャを声色スコアに連動
- **キャラごとに「拾う声色」を変える**: クロウ（寡黙）は怒りに、リリア（魔女）は嘘に敏感、みたいな個性付け
- **長期記憶**: 過去の会話を埋め込みベクトルで保持、「先週は元気そうだったのに今日は…」みたいな比較反応

「**声色で AI の反応が変わる**」というだけで、ゲーム・教育・福祉・ビジネスに応用余地が広がる感触があります。テキスト LLM 全盛の時代に、**声というモダリティをもう一段深く読み解ける**のは AmiVoice の感情分析ならではの強みだなと感じました。

---

## まとめ：AmiVoice 感情分析を使うときの 4 つの注意点

| # | 注意点 | 症状 | 対策 |
|---|--------|------|------|
| 1 | **非同期 API はポーリング必須** | 認識結果が常に `"..."` | 投入 → GET ポーリング → completed で結果取得 |
| 2 | **値範囲はパラメータごとに違う** | 判定が永遠に発動しない | 0-30 範囲は `>0` で有意とみなす |
| 3 | **短い録音では energy 単独判定不可** | 声色を変えてもスコア不変 | 30 秒以上の録音 + 補助指標を最優先 |
| 4 | **生成 AI に指標名を渡すと喋る** | 「エネルギーが低いね」 | 日本語化 + 禁止用語の明示 |

AmiVoice の感情分析（ESAS）は、ES ジャパンの最新エンジンを使った非常に解像度の高いツールですが、その特性を理解せずに使うと「常に悲しい判定」のような事故に直面します。

逆に、これらの落とし穴を踏まえれば、**「声色で意味が変わる対話 AI」** のような、テキストだけの LLM では実現できない新しい音声体験を作るのに最適な API です。
